> **POC-A 핵심 실측(13장 상세):** 표준화 8변수 PCA에서 PC1(크기 인자)이 분산의 **72.0%**를 설명. edit_churn은 크기와 ρ=0.81~0.94, W2는 ρ=−0.78~−0.85(역상관)로 **크기에 종속**. W3만 상대적으로 크기-독립적(ρ 0.51~0.62)이나 median=0으로 sparse. → 무료 축은 크기 보정 없이는 "낭비"가 아니라 "세션 규모"를 잰다.

> **2차 검증 반영(v0.2.1):** 잔차화 조작적 정의(3.2a: size 변수·회귀·영과잉 처리), W2 산식(3.2), 검증 임계(10장 잠정 수치), 마일스톤 순서(9장 Pilot-0→M0)를 추가해 **구현 가능·반증 가능 수준**으로 보강. 모든 잠정 임계는 Pilot-0 실측으로 확정한다.

## 0. 이 문서의 한 줄 요약

같은 결과를 만드는 데 **꼭 쓰지 않아도 됐을 토큰·반복·재작업**이 얼마였는지를, **평가 시스템 자체가 토큰을 거의 쓰지 않으면서** 정량적으로 보여주려는 도구의 설계.

핵심 설계 원리 세 가지를 먼저 못 박는다.

1. **측정 대상은 "회피 가능한 낭비"지만, 무료 결정론 층은 그것을 직접 재지 못한다.** 무료 신호는 "총 폐기·재탐색·반복"의 *크기 보정 잔차*를 잴 뿐이고, "회피 가능" 여부(반사실 판정)는 기준선과 LLM 판별이 있어야 한다. v0.1은 이 둘을 혼동했다 → 무료 축에서 "회피 가능"이라는 단어를 떼고 "크기보정 낭비 신호"로 정직하게 표기한다. [C1·C5]
2. **측정의 상당부는 결정론으로 가능하나, "80%"는 검증 전제 조건부다.** 결정론으로 보이는 지표 다수(W1·W4·생존율)는 PR↔세션 attribution에 의존하는데 그 연결이 결정론적으로 불가능하다. 따라서 무료 축은 attribution 없이 계산 가능한 것(단일세션 내 신호)으로 한정한다. [C2·B1]
3. **이 도구는 실행 효율만 측정하며, 방향의 타당성은 측정하지 않는다.** 잘못된 방향으로 정확하게 일한 세션은 이 도구에서 높은 점수를 받을 수 있다(false positive). 모든 산출물에 이 면책을 명시한다. [C5]

## 1. 제품 철학과 문제 정의

### 1.1 측정 대상의 재정의

기존 생산성 도구는 산출물(PR 수·LOC) 또는 투입(토큰·시간)을 측정한다. 이 도구는 **둘 사이의 간극, 즉 "결과로 남지 못하고 버려진 투입"** 을 측정하려 한다.

> 정의 — **회피 가능한 낭비(Avoidable Waste):** *그 시점에 이미 확보 가능했던 정보를 고려했을 때, 일어나지 않아도 됐던 토큰·반복·재작업.*

**정의의 두 가지 보강(v0.2):**

- **"확보 가능했던 정보"의 주체를 명시한다.** [M3] 케이스마다 주체가 달라 정의가 흔들렸다. 주체를 세 층으로 구분한다: (a) 사용자가 알 수 있었던 정보(예: 줘야 했던 파일 맥락), (b) Claude가 컨텍스트에서 이미 알 수 있었던 정보(예: 재탐색), (c) 규범적 이상 협업이라면 확보 가능했을 정보. 각 판별 케이스(2.3)는 어느 주체 기준인지 태그한다.
- **"회피 가능"은 반사실(counterfactual)이며 기준선(baseline)을 요구한다.** [C5] "일어나지 *않아도 됐을*"을 판정하려면 "합리적 최소 궤적"이라는 기준선이 있어야 한다. 기준선 없이 결정론만으로는 "회피 가능"을 판정할 수 없다. 무료 층은 기준선 없이 계산되므로 "회피 가능"이 아니라 "총 폐기/재탐색의 크기보정 잔차"를 측정한다. 기준선 정의는 12장 미해결 질문으로 남긴다.

### 1.2 명시적 비목표 (Non-Goals)

- ❌ **인사고과·승진·보상 입력값.** 측정이 목표가 되면 지표는 죽는다(Goodhart). AI 기반 고용 평가는 규제 고위험.
- ❌ **개인 간 순위(leaderboard).**
- ❌ **"역량"·"실력" 점수 자동 산출.**
- ❌ **감시 도구.** 기본은 개인이 자기 데이터를 자기만 보는 셀프 도구.
- ❌ **방향·전략의 타당성 평가.** [C5] 이 도구는 "어떻게 일했나"의 효율만 본다. "옳은 것을 했나"는 범위 밖이다.

**비목표를 지키기 위한 구조적 가드(v0.2):** [H5] v0.1은 "고과 미연동"(비목표)과 "팀 회고 집계"(목표)를 병기해 충돌했다. 소규모 팀에서 패턴 식별은 곧 개인 식별이 되어 사실상 고과 전용이 발생한다. 따라서 **팀 집계는 M3의 별도 모듈로 분리**하고, k-익명성(팀 규모 하한·노출 단위) 또는 차분 프라이버시를 기술적으로 강제하기 전에는 제공하지 않는다. M0~M2는 완전 셀프 전용으로 고정한다. 라이선스 조항은 강제력이 없으므로 기본값·아키텍처로 막는다.

### 1.3 목표 (Goals)

- ✅ 개인이 "나는 같은 결과를 더 적은 헛수고로 낼 수 있었나?"를 **시계열로** 본다. → 시계열은 **영속 저장**을 전제한다(H1, 4장·9장 참조).
- ✅ 팀이 회고에서 "어디서 반복적으로 노력이 새는가?"를 집계로 본다 — **단, M3 별도 모듈 + 익명성 기술강제 이후에만**.
- ✅ AI 협업 스킬의 개선 여지를 구체 사례로 짚는다.
- ✅ 위를 **세션 토큰의 1~2% 미만** 비용으로 달성한다 — 단, 이 비율은 *실행 시* 평가 토큰만 센다. 도구 자체의 구현·검증 비용은 ROI 섹션(13장)에서 별도로 다룬다. [H8]

### 1.4 측정 윤리 원칙

1. **자기 대비만, 타인 대비 금지.**
2. **단일 점수 금지, 프로파일 제공.** 단, 프로파일 축들이 실제로 독립적이어야 의미가 있다(13장 다중공선성 점검 참조).
3. **불확실성 동반 표기.** 모든 추정치에 신뢰구간/근거. "측정 불가"를 표현할 수 있어야 한다. 특히 **"0 = 낭비 없음"인지 "0 = 데이터 없음"인지** 구분하는 데이터품질 플래그를 모든 지표에 단다. [A5·C3]
4. **점수가 아니라 사례로 말한다.**

## 2. 핵심 개념 정의

### 2.1 작업 단위 (Work Unit, WU)

평가의 최소 단위. v0.1은 WU를 "PR + 세션"으로 정의하고 PR↔세션 연결을 결정론적이라 했으나, **이 연결은 결정론적으로 불가능하다**(Claude Code에 commit/PR에 session_id를 박는 공식 메커니즘이 없음). [C2]

**v0.2 WU 정의 — 2단 구조:**

```
WU(기본, 단계1)  = 단일 세션                      // attribution 불필요, 항상 계산 가능
WU(확장, 단계2)  = PR + 그것을 만든 세션(들)        // attribution 커버리지 검증(Pilot-0) 후 조건부
```

- **단계1(단일 세션):** 모든 무료 신호는 단일 세션 경계 안에서 계산한다. attribution이 없어도 항상 성립.
- **단계2(PR 단위):** Pilot-0에서 attribution 커버리지(커밋/PR에 세션을 역추적 가능한 비율)와 cwd+시간창 휴리스틱의 오귀속률(특히 병렬 에이전트 환경)을 실측한 뒤에만 활성화. **오귀속률 > 10%(잠정)면 단계2 보류.** [NM-2]
- 멀티세션·멀티PR WU 경계 규칙은 12장 미해결 질문으로 유지.

### 2.2 효율 프로파일 (Efficiency Profile)

각 WU는 단일 점수가 아니라 여러 축의 프로파일로 표현된다. **단, 모든 무료 축은 크기-잔차로 정의한다(13장)** — 그렇지 않으면 축이 "세션 규모"의 대리지표가 된다. [C1·C4]

| 축 | 의미 | 단계 | 측정 비용 |
|---|---|---|---|
| **컨텍스트 신호(Context)** | 캐시 무효화 이벤트 등 컨텍스트 비효율 (W2 재정의, 크기보정) | 1 | 결정론 |
| **재탐색 신호(Re-research)** | 동일 대상 반복 read의 크기보정 잔차 (W3, 영과잉 처리) | 1 | 결정론 |
| **편집 반복 신호(Edit-thrash)** | 동일 파일 반복 편집의 크기보정 잔차 (edit_churn) | 1 | 결정론 |
| **생존율(Survival)** | 산출물 중 머지까지 살아남은 비율 | **2(조건부)** | attribution 의존 |
| **폐기 신호(W1·W4)** | 폐기 생성·폐기 작업물 | **2(조건부)** | attribution 의존 |
| **재작업 성격(Rework type)** | 반복이 "새 정보형"인가 "회피 가능형"인가 | 2 | LLM(샘플링) |
| **난이도(Difficulty)** | 정규화용 보정 계수 (점수 아님, 회귀 잔차 방식) | 1~2 | 프록시 + 저빈도 LLM |

> **불확실성·데이터품질 라벨:** 각 축은 `value ± band` + `data_quality{otel?, hook?, git?}` + `is_zero_real(bool)`로 보고. 분산이 좁은 축(예: W2)은 "변별력 낮음"으로 명시 라벨. [C3·H7]

### 2.3 "낭비" vs "생산적 반복" 판별

v0.1 판별표는 다수 케이스를 "결정론"으로 분류했으나, 실제로는 숨은 휴리스틱/NLP 의존이 있고 일부는 현재 수집 파이프라인에서 데이터가 버려진다(claude-alive가 tool_input/tool_response를 받고도 폐기). v0.2는 각 케이스에 **(판별 주체, 실제 판별 방법, 데이터 가용성)**을 명시하고 **"판별 불가(ambiguous)" 카테고리를 정식 추가**한다. [D1·D2]

| 케이스 | 분류 | 실제 판별 방법(정직판) |
|---|---|---|
| 테스트가 드러낸 버그로 재작업 | 생산적 ✅ | 테스트 실패 이벤트 = **상관 힌트**(인과 아님), 모호 시 LLM |
| 새 엣지케이스 발견 후 보강 | 생산적 ✅ | 결정론 힌트 + LLM 확인 |
| 첫 프롬프트 모호 → 재생성 | 낭비 ❌ | 동일 타깃 재생성 + 새 정보 없음 → **LLM 판별 필요** |
| 기존 맥락 안 줘서 실패 후 재시도 | 낭비 ❌ | 후행 프롬프트의 파일 맥락 추가 여부 → 명시 경로면 결정론, **암묵 언급이면 NLP/LLM** |
| 거대 무관 컨텍스트 주입 | 낭비 ❌ | 입력 토큰↑ + 캐시 적중↓ + 미참조 → 결정론(단 W2 크기보정 필요) |
| 동일 파일 N회 재탐색 | 낭비? | `tool_use` 카운트 → 결정론, **단 tool_input 수집 선행 필요**(현재 버려짐) |
| 위에 해당 안 됨 | **ambiguous** | LLM 표본 판별 또는 "측정 불가" |

판별의 1차 분류가 결정론으로 가능한 비율(=LLM까지 올라가지 않는 비율)은 **가정이 아니라 Pilot-0/M2에서 실측**한다.

## 3. 측정 모델 (정량화 설계)

### 3.1 시그널 분류와 3소스 조인

**소스가 셋이고, 결정론 계산조차 조인을 요구한다.** [G1·H3] 토큰/비용은 OTel 또는 transcript JSONL에서, 프롬프트/도구는 훅에서, 머지/revert/diff는 git에서 온다. 이들을 `session.id`로 조인해야 W2조차 계산된다. 따라서 L1에 **명시적 조인 레이어**를 둔다.

- **토큰 출처 주의:** OTel `claude_code.token.usage`(input/output/cacheRead/cacheCreation) 또는 transcript JSONL 파싱. **JSONL은 비공식·변동 포맷·30일 TTL**이고 이 레포는 이미 파싱 크래시 전례가 있다 → 파싱은 방어적으로, 영속 저장으로 TTL을 보완. [F2·R4]
- **조인 키 누락 대비:** OTel `session.id`는 선택 속성. 누락 WU는 `data_quality`로 표기하고 측정 불가 처리.

**무료(결정론) — 단일 세션 내에서 계산 가능:** 토큰 분해, 캐시 적중률, `tool_use` 카운트(재탐색·편집 반복), 프롬프트/턴 수.
**조건부 무료 — attribution 필요:** 머지 여부·revert·폐기 브랜치·생성 대비 머지 라인(W1·W4·생존율).
**유료(LLM, 최소화):** 반복 성격 분류, 난이도 보정, 회피 가능성 판정.

### 3.2 낭비 지표 카탈로그 (크기-잔차 재정의)

각 지표는 원자료(raw)로 저장하고, **점수화 시 크기 인자에 회귀시킨 잔차**를 사용한다. [C1·C4·13장]

```
W2_raw 재무효화토큰 = Σ cache_creation_input_tokens − 첫_어시스턴트_메시지의_cache_creation
                     (초기 워밍업=회피불가 제외. 세션 중반 이후 거대·변동 컨텍스트
                      재주입으로 캐시가 반복 무효화된 토큰을 포착)
W2 컨텍스트 신호  = residual( W2_raw | size )                      [단계1]
W3_raw           = 동일 path 중복 read 수
W3 재탐색        = residual( W3_raw | size ), 허들 2부 모델         [단계1, 영과잉]
                   (셋 중 가장 크기-독립적이나 median=0 → is_zero_real 필수)
WC_raw           = 동일 파일 반복 편집 수
WC 편집 반복     = residual( WC_raw | size ), 허들 2부 모델          [단계1, 영과잉]
                   (raw는 size와 ρ=0.94 → 잔차화 전 낭비축 부적합)
W1 폐기 생성     = (기준선 대비 초과 생성 라인) / 생성 라인          [단계2, attribution]
W4 폐기 작업물   = 폐기 브랜치·revert 토큰 / 총 토큰                [단계2, attribution]
W5 회피가능 재작업 = 회피형 분류 반복 토큰 / 총 반복 토큰            [유료, LLM]
```

> W1의 분모 "생성된 라인"은 tool_input 없이 정의 불가하므로(현재 버려짐), **tool_input 수집을 선행 조건**으로 명시하고 산식을 코드 수준으로 확정한다(M0 명세 단계). [F7]

### 3.2a 잔차화 조작적 정의 [C1·NM-2·NM-5]

선언이 아니라 구현 가능한 수준으로 고정한다.

- **size 변수:** `size = log(total_tokens)` (단일 크기 인자, 13장 PCA의 PC1 대용). Pilot-0에서 log(tool_calls) 등 대안과 설명력 비교 후 확정.
- **회귀 방법:** 순위기반/로버스트 회귀(Theil–Sen 또는 rank 잔차) — 비정규·이상치에 강건. OLS 금지(POC-A 분포가 비정규).
- **영과잉 지표(W3·WC) — 허들 2부 모델:** ① `P(신호>0)`(발생 여부) ② 신호>0일 때 크기보정 크기. 대시보드는 두 부분을 분리 표기하고, `is_zero_real`로 "0=발생안함 vs 0=데이터없음"을 구분.
- **잔차 부호 규약:** 양(+) 잔차 = 같은 크기 대비 신호 초과(낭비 의심), 음(−) = 미만.

### 3.3 난이도 정규화 (순환논리 제거)

v0.1은 난이도 프록시(파일수·diff 분산도)와 낭비 신호(생성 라인·재탐색)가 **같은 작업 규모에서 파생**되어, 나눗셈 정규화가 신호를 자기상쇄했다. [C4·CV-3]

**v0.2 수정:**

- **난이도 프록시는 작업 *시작 시점에 고정되는* 양만 사용:** 요구 명세 길이, 대상 모듈의 사전 순환복잡도, 의존 그래프 깊이, 신규 vs 수정. 작업 *진행 중* 발생량(diff 분산·폐기 라인)은 **난이도에서 배제**(그것은 낭비 측이다).
- **나눗셈 대신 회귀 잔차 정규화:** `waste ~ difficulty` 회귀 후 잔차를 사용. 분모-0 폭발과 상쇄를 동시 회피.
- **정규화 전후 신호 보존량**(잔차 분산 / 원시 분산)을 검증 지표로 추가. 보존량이 임계 이하면 정규화가 신호를 죽인 것으로 간주.

### 3.4 점수화와 불확실성

- 각 축: 같은 사용자의 과거 분포 대비 변화. **분포 형태를 진단**해 변환 결정(영과잉은 영과잉 모델, 좁은 분포는 "변별력 낮음" 라벨). 백분위는 0-과잉/협소 분포에서 불안정하므로 **변화점 탐지(부트스트랩 신뢰구간)**를 우선. [CV-2]
- 프로파일 카드는 축별 막대를 **상호 비교 가능하게 병치하지 않는다.** 이질 단위·분포라 막대 길이 비교가 아티팩트가 된다 → "자기 기준선 대비 절대 변화 + 신뢰구간"으로 축별 독립 표기. [CV-4·H7]
- 단일 종합점수 금지.

## 4. 시스템 아키텍처

**v0.2 핵심 결정: Efficio는 claude-alive의 확장이 아니라 별도 제품이다.** [H2·CV-6] claude-alive는 휘발성 라이브 뷰(인메모리 SessionStore, 완료세션 50개·이벤트 1000개 슬라이딩 윈도우, 영속 저장 없음)다. Efficio는 영구 분석 저장소·git 수집기·OTel 파이프라인을 요구하므로 수명주기·데이터 모델이 다르다. **두 도구는 동일 훅 이벤트 소스를 공유하는 별도 소비자**로 느슨하게 결합하고, 코드 재사용은 타입 정의·`toolMapper`·FSM 수준에 한정한다. SessionStore 자체는 재사용 부적합.

```
┌─────────────────────────────────────────────────────────────┐
│  L0  수집                                          토큰: 0   │
│   - OTel exporter(토큰/비용)  - 훅(프롬프트/도구)  - git 메타 │
├─────────────────────────────────────────────────────────────┤
│  L1  조인 + 특징 추출                               토큰: 0   │
│   - ★ session.id 기반 3소스 조인 + data_quality 플래그       │
│   - WU 조립(단계1=단일세션 기본 / 단계2=PR 조건부)            │
│   - 크기-잔차 신호 W2/W3/WC, 난이도 프록시(시작시점 고정량)   │
│   - 모호도 플래그 산출 → L2 게이트                            │
├─────────────────────────────────────────────────────────────┤
│  L2  판별 (Adjudication)                       토큰: 최소·게이트│
│   - 플래그된 WU만, 압축 입력, 작은 JSON 출력                  │
├─────────────────────────────────────────────────────────────┤
│  L3  영속 저장 (★ M0 필수)                                   │
│   - SQLite: WU 원자료 + 결정론 지표 영구 저장(재계산 가능)    │
│   - 30일 TTL 소스에 의존하지 않도록 실시간 append            │
├─────────────────────────────────────────────────────────────┤
│  L4  리포팅                                    토큰: 0       │
├─────────────────────────────────────────────────────────────┤
│  L5  자기 관측: 평가비용 비율 + gate_suppression_rate        │
└─────────────────────────────────────────────────────────────┘
```

> 메트릭/훅 명칭과 transcript 스키마는 버전에 따라 변동 가능하므로 수집 어댑터로 추상화하고 명칭을 버전 핀한다. [R4]

## 5. 토큰 효율 설계

> 원칙: "측정은 공짜가 기본값, LLM은 청구해야 하는 예외."

### 5.1 계단식 평가 (Tiered Evaluation)

| Tier | 무엇 | 발동 비율 | WU당 토큰(예시) |
|---|---|---|---|
| **T0** 결정론 | 단계1 신호, 난이도 프록시 | 100% | **0** |
| **T1** 경량 분류 | 난이도 보정 | 실측 예정 | ~1.5~2k |
| **T2** 판별 | 반복 성격 분류 | 실측 예정 | ~3~5k |
| **T3** 감사 | 표본 검증·드리프트 | ~1~5% | 주기적 |

발동 비율은 **게이트 알고리즘에 의해 결정되며, 게이트가 명세되기 전에는 비율을 약속하지 않는다.** [F5]

### 5.2 게이트(모호도 플래그) — M1 선결 [F5]

게이트는 시스템의 경제적 핵심이나 v0.1에서 미명세였다. **M1 진입 전 다음을 완전 명세한다:** 플래그 조건 목록 · 각 조건 임계값 · 복수 조건 AND/OR 결합 · Tier 상승 결정 로직. 이후 Pilot-0/POC 데이터로 T1/T2 실제 발동률을 측정해 SLO 계산을 검증한다.

### 5.3 입력 압축 / 출력 최소화

- 원자료를 모델에 넣지 않는다. diff stat + 대표 hunk 1~3개 + 결정론 수치만. 세션 궤적은 구조화 요약만.
- 모델 출력은 작은 JSON(라벨/점수). 자연어 설명은 결정론 지표 + 템플릿으로 L4에서 조립(토큰 0).

### 5.4 캐싱·배치·샘플링

프롬프트 캐시(고정 루브릭), WU 단위 1회 평가(L3 영구 저장, 재계산 금지), 배치 API, 캘리브레이션 표본 감사.

### 5.5 자기 토큰 예산 (Self-Budget SLO)

```
eval_cost_ratio       = (평가 토큰) / (측정 대상 세션 토큰)
gate_suppression_rate = (게이트 보수화로 평가 억제된 모호 WU) / (전체 모호 WU)   ★신규
목표 SLO: eval_cost_ratio median ≤ 1%, p95 ≤ 2%
```

**Silent degradation 차단:** [F6] "초과 시 게이트 자동 보수화"는 측정 품질을 조용히 떨어뜨리면서 SLO는 건강해 보이게 한다. 따라서 (a) `gate_suppression_rate`를 L5에서 함께 추적·노출하고, (b) 자동 보수화 임계 상승에는 **시간기반 자동 원복(예: 24h)**을 둔다. "비용 1% 미만" 약속은 "최소 N% WU에서 판별 시도 완료" 조건과 함께 표기한다.

> **SLO 민감도(정직판):** [C1-SLO] v0.1 워크드 예시(2.0%)는 p95 천장과 같다. median ≤1%는 "발동 WU < 50%"를, p95 ≤2%는 "상위 꼬리에서 T1+T2 동시 발동이 드묾"을 전제한다. low/mid/high 발동률 시나리오별 SLO 달성 조건을 정량 분석해 명시한다(게이트 명세 후).

## 9. MVP 범위와 마일스톤

> **마일스톤 순서(엄격):** Pilot-0 → **M0 범위확정**(Pilot-0 결과로 가부·축 결정) → M0 구현 → M1 → M2 → M3. M0는 Pilot-0 게이트를 통과해야 시작하는 **조건부 단계**이며, Pilot-0와 병렬이 아니다. [NM-3]

**Pilot-0 — 전제 실측 (★신규, M0 전 게이트):** [C2·C3·종합]
(a) attribution 커버리지(세션↔커밋 역추적 가능 비율)와 cwd+시간창 휴리스틱 오귀속률(병렬 에이전트 포함) 실측 → 단계2 가부 결정(오귀속률 ≤10% 잠정). (b) **신호 밀도 실측**(비-0 WU 비율; W3·WC 각각) → 비-0 WU ≥30%(잠정)면 해당 축 M0 채택, 미달이면 단계1 핵심 축에서 제외. (c) 더 크고 균질한 표본으로 구성타당도(13장) 재현(PC1 ≤50% 잠정). (d) 잠정 임계 4종을 데이터로 확정.

**M0 — 단일세션 결정론 코어 (LLM 0%):** OTel/훅 수집 → L1 조인 → **Pilot-0 통과한 단계1 크기-잔차 신호** → **영속 저장(SQLite, 필수)** → 개인 프로파일 카드 + 시계열. 가치 정의를 "단일세션 크기보정 효율 프로파일 + 영속 시계열"로 한정. **Pilot-0(b)에서 채택된 축이 0개면 M0를 진행하지 않고 3.3 미해결 질문(신규 결정론 신호)으로 회귀한다.**

**M1 — 게이트 + T1:** 게이트 알고리즘 완전 명세(5.2) → 난이도 보정 → 자기 토큰 SLO + gate_suppression_rate 계측.

**M2 — T2 판별:** 반복 성격 분류, ambiguous 비율 실측, 사례 기반 피드백.

**M3 — 팀 집계 + T3 감사:** **k-익명성/차분 프라이버시 기술강제 이후에만.** 캘리브레이션·드리프트 모니터.

> M0가 "출시 가능한 가치"인지는 Pilot-0(b) 신호 밀도 실측이 결정한다. 밀도가 임계 미달이면 M0 범위를 재정의한다(공백 대시보드 출시 방지). [F1]

## 10. 검증 계획 (정량 임계 명시)

v0.1의 "상관 유의"·"κ 충분"은 효과크기 임계가 없어 반증 불가능했다. v0.2는 사전등록 수준의 정량 임계를 둔다. [H6·CV-7]

| 가설 | 검증 방법 | 합격 기준(정량) |
|---|---|---|
| H1: 잔차 낭비 지표가 실제 헛수고와 상관 | 잔차 지표 vs 전문가 수기 라벨(2인+, IRR κ 보고) | **Spearman \|ρ\|≥0.5** + 다중비교 Bonferroni 보정 |
| H1b: 신호 밀도 충분 | 비-0 WU 비율(축별) | **≥ 30%(잠정, Pilot-0 확정)** [C3] |
| H2: 난이도 정규화가 어려운 작업 처벌 안 함 + 신호 보존 | 동등성 검정(TOST) + 잔차/원시 분산비 | 편향 부재 입증(검정력 ≥0.8) + **보존량 ≥ 0.5(잠정)** |
| H3: 게이트가 토큰 절약 | eval_cost_ratio 분포 | median ≤1%, p95 ≤2% |
| H4: T2 분류가 전문가와 일치 | 모호 표본 vs 인간 라벨 | **κ≥0.6** (95% CI 하한 ≥0.4) |
| H5: 구성타당도 — 축이 크기 대리지표 아님 | W지표 상관행렬·PCA(13장) | **잔차화 후 크기상관 \|ρ\|<0.3** (필수) AND **PC1 분산설명 ≤50%(잠정)** |

> **선행 검증:** H1·H1b·H5가 깨지면 전체 전제가 무너진다. Pilot-0/M0 직후 우선 검증. POC-A(13장)는 H5에 대한 1차 경고를 이미 제공했다.

## 11. 리스크 레지스터

- **R1 난이도 오추정 → 어려운 작업 불이익.** 완화: 회귀 잔차 정규화 + 잔차 노출 + 불확실성.
- **R2 결정론 지표가 헛수고와 무상관.** 완화: H1 선행 검증.
- **R3 비목표 침식(고과 전용).** 완화: 라이선스 + 기본값 + M3 익명성 기술강제. [H5]
- **R4 텔레메트리 스펙 변경.** 완화: 어댑터 추상화 + 명칭 버전 핀.
- **R4b 30일 TTL로 역사적 시계열 소실.** [F2·H1] 완화: 실시간 수집 → SQLite 영속화(M0 필수).
- **R5 평가 도구 토큰 폭주.** 완화: SLO 가드 + 게이트 보수화 — **단 R5 완화가 R6를 유발**.
- **R6 Silent degradation(보수화가 측정 품질 저하).** [F6] 완화: gate_suppression_rate 추적 + 시간기반 원복.
- **R7 구성타당도 붕괴 — 축이 크기 대리지표.** [C1·C4·13장] 완화: 크기-잔차 재정의 + PCA 게이트.
- **R8 신호 밀도 0 — 공백 대시보드.** [C3·F1] 완화: Pilot-0(b) 밀도 실측 후 M0 범위 결정.
- **R9 방향오류 낭비 false positive.** [C5] 완화: 면책 명시(범위 밖).
- **R10 프라이버시(프롬프트 원문).** 완화: 로컬 우선 + 비저장 옵션.

## 13. 구성타당도 — POC-A 실측 (부록)

**질문:** 무료 W지표가 잠재변수 "회피 가능한 낭비"를 재는가, 단일 "세션 크기" 인자로 환원되는가?
**방법:** ~/.claude/projects 세션 transcript 40개를 0토큰으로 파싱, Spearman 상관 + PCA. 스크립트: `docs/poc/poc_a_waste_signals.py`, `docs/poc/poc_b_construct_validity.py`.

**결과:**
- 표준화 8변수 PCA에서 **PC1(크기 인자)이 분산의 72.0%** 설명. edit_churn(−0.39)이 크기 프록시(0.38~0.41)와 동일 강도로 PC1에 적재.
- 후보 낭비지표 × 크기(Spearman ρ): W2 −0.78~−0.85(역상관), W3 0.51~0.62, **edit_churn 0.81~0.94**.
- 낭비지표 3개만의 PCA: PC1 50.7% → 서로 완전 중복은 아님(~3차원 잔여 구조).

**해석:** W2·edit_churn은 크기에 종속(C1·C4 확인). W3은 가장 크기-독립적이나 sparse(C3). → **크기 보정 잔차가 필수**이며, 보정 전 무료 축은 "세션 규모"의 대리지표.

**한계:** n=40, 1턴 보안리뷰 + 거대 도구개발 세션이 섞인 이중봉 표본이라 크기 인자가 부풀려졌을 수 있음. **확정 전 더 크고 균질한 표본으로 재현 필요(Pilot-0)**. 방향성 스모크테스트로는 우려를 지지하나 단정 근거는 아님.

### 13.1 Pilot-0 실측 결과 (n=184 균질 개발세션)

스크립트: `docs/poc/pilot0_harness.py`. 표본: turns≥3·assist≥4(1턴 리뷰 제외), median 8턴.

| 측정 | 결과 | 잠정 임계 | 판정 |
|---|---|---|---|
| (a) attribution 오귀속률 | **55.7%** (커버리지 63%, 충돌 275/494) | ≤10% | **FAIL** → 단계2(PR WU) 보류, WU=단일세션 확정 |
| (b) 신호 밀도 비-0 비율 | W2 100% / W3 52.2% / WC 62.5% | ≥30% | **PASS(3축)** → M0 채택 |
| (c) 구성타당도 raw PC1 | **78.8%** (크기 지배) | ≤50% | FAIL(raw) — 잔차화 필수 |
| (c) 잔차화 후 크기상관 ρ | W2 0.10 / W3 0.13 / WC 0.12 | <0.3 | **PASS** |
| (c) 신호 보존량(잔차/원시 분산) | W2 0.72 / W3 0.79 / WC 0.66 | ≥0.5 | **PASS** |

**해석:**
- **attribution은 단계2를 막는다(확정).** 오귀속률 55.7%는 사용자가 동일 repo에서 세션을 다수 병렬 실행하기 때문(claude-alive 사용 패턴과 정합). → W1·W4·생존율은 M0에서 제외하고 단일세션 WU로 간다는 v0.2.1 결정이 데이터로 확인됨. [C2]
- **신호 밀도는 충분(확정).** POC-A에서 W3/WC median=0이던 건 1턴 리뷰 혼입 탓. 개발세션으로 한정하면 3축 모두 ≥52% 비-0. M0가 보여줄 신호는 있다. [C3 해소]
- **잔차화가 실제로 작동(확정).** raw는 크기 지배(PC1 78.8%, raw ρ 0.74~0.87)지만, Theil–Sen 잔차화 후 크기상관 0.10~0.13으로 떨어지고 **분산 66~79%가 보존**됨 → 크기를 제거해도 신호가 남는다. C1/C4가 선언이 아니라 실측으로 처리 가능. [C1·C4 해소]

**남은 한계(중요):**
- **구성타당도 ≠ 기준타당도.** 잔차가 "크기가 아님"은 보였으나 "실제 헛수고임"(H1)은 아직 미검증 — 라벨이 필요(POC-A 라벨링 보류분). 잔차축이 진짜 낭비를 추적하는지는 별도 확인 대상.
- 잔차 ρ<0.3은 Theil–Sen 특성상 부분적으로 구조적 — 핵심 근거는 **보존량(66~79%)**.
- 표본이 사용자 프로젝트에 편중(l2u-2nd 58/184). 타 사용자 일반화는 별도 표본 필요.

### 13.2 H1 기준타당도 — 잔차 신호 vs 사람 라벨 (2라운드)

스크립트: `docs/poc/h1_labeling.py`·`h1_correlate.py`(R1), `h1b_labeling.py`·`h1b_correlate.py`(R2). 임계 Spearman ρ≥0.5.

**라운드1 (n=16, 러프 라벨, 결과/체감 기준):** 합성 ρ=−0.29, W2 −0.05, W3 −0.15, WC −0.50 (전부 FAIL). 라벨이 raw tokens(+0.34)와 더 상관 → 사람이 "크게/오래 걸린 것"을 헛수고로 인식. **라벨 정의가 지표와 불일치.**

**라운드2 (n=30, 라벨 기준 정렬="기계적 헛수고만, 결과/방향 배제", 1=매끄러움..5=많음):**

| 신호 | ρ | 95% CI(부트스트랩 B=3000) | P(ρ>0) | P(ρ≥0.5) |
|---|---|---|---|---|
| **W2 단독(재무효화)** | **+0.57** | [+0.18, +0.82] | 99.8% | 66% |
| W3 단독(재탐색) | +0.12 | [−0.26, +0.47] | 72% | 1% |
| WC 단독(편집반복) | +0.23 | [−0.16, +0.58] | 88% | 8% |
| 합성(등가중) | +0.43 | [+0.06, +0.71] | 99% | 35% |
| 합성(W2우위 2:1:1) | +0.54 | [+0.21, +0.77] | 99.8% | 60% |
| (참고) raw tokens/turns/tools | +0.02~0.13 | — | — | — |

**결론:**
- **방향 확립:** 잔차 신호(특히 W2)는 체감 기계적 낭비와 **양의 상관이 견고**(P(ρ>0)≈99.8%). 라운드1 음의 상관은 **라벨 정의 불일치 산물**이었음 — 기준을 기계적 헛수고로 정렬하니 부호가 반전.
- **크기 분리 확인:** 정렬 라벨은 raw 크기와 무상관(ρ 0.02~0.13) → 지표가 크기가 아닌 낭비를 잡음.
- **임계는 미확정:** ρ≥0.5를 W2(0.57)·W2우위합성(0.54)이 점추정으로 넘으나 CI 하한 0.18~0.21·P(ρ≥0.5) 60~66%로 **견고히 넘는다 단정 못 함.** W3·WC는 개별 기준타당도 약함 → 가치가 **W2(컨텍스트 재무효화)에 집중.**

**설계 함의:** 프로파일은 **W2 우위**로 구성(등가중 금지). W3·WC는 실험축으로 표기. 헤드라인은 보수적으로("W2가 체감 기계적 낭비를 추적, ρ≈0.5, n=30 파일럿"). 임계 견고 통과엔 더 큰 n + 전체 1~5 척도 라벨 필요(이번 라벨은 1~3에만 분포해 상한이 눌림).

**한계:** n=30, 라벨이 1~3에 몰려(4~5 미사용) 동점 많고 검정력·상한 제약. 단일 사용자·자기 프로젝트. ρ는 점추정이며 CI가 넓음.
